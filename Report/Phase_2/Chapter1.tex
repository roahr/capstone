\chapter{\uppercase{Introduction}}
\pagenumbering{arabic}
\setcounter{page}{1}

Software vulnerabilities cost the global economy an estimated \$2.4~trillion annually, yet the tools charged with finding them produce noise at industrial scale. In 2023, the National Vulnerability Database catalogued over 25,000~new CVEs, a 15\% year-over-year increase~\cite{nvd2023}. Static Application Security Testing (SAST) tools flag between 68\% and 95\% false positives~\cite{ghostsecurity2025, siftingnoise2026}. Machine learning models trained to improve on SAST collapse from 93\% F1 on curated benchmarks to single-digit F1 on realistic data~\cite{realvul2024, primevul2024}. Large Language Models (LLMs) hallucinate vulnerabilities at rates approaching 90\%~\cite{llm4vuln2024}. No single analysis modality is trustworthy in isolation.

We present a multi-stage code security framework that treats vulnerability detection as a \textit{triage} problem rather than a classification problem. Instead of running every finding through every analysis stage, the cascade routes each finding to the cheapest stage capable of resolving it:

\begin{itemize}
    \item \textbf{Stage~1 (SAST):} Tree-sitter pattern matching and CodeQL taint analysis with 4-factor uncertainty scores. Cost: $<$100\,ms, free.
    \item \textbf{Stage~2 (Graph):} Joern Code Property Graph, backward slicing, GraphCodeBERT embeddings, GNN classification, and Adaptive Prediction Sets (APS) conformal prediction. Cost: 1--3\,s, free (local).
    \item \textbf{Stage~3 (LLM):} Dual-agent adversarial analysis (attacker + defender) with Retrieval-Augmented Generation (RAG) knowledge retrieval, consensus engine, and Common Vulnerability Scoring System (CVSS)~v3.1 scoring. Cost: 5--15\,s, API fees.
    \item \textbf{Stage~4 (Fusion):} Common Weakness Enumeration (CWE)-adaptive weighted score combination, three-tier classification, and Static Analysis Results Interchange Format (SARIF)~2.1.0 report generation. Cost: $<$10\,ms.
\end{itemize}

A 4-factor uncertainty score governs escalation: only findings where the current stage is measurably uncertain advance to a more expensive analysis. Evaluation on 15 open-source repositories shows that 85\% of findings resolve at Stage~1, 2\% at Stage~2, and 13\% at Stage~3, with zero findings left unresolved. For a scan producing 1,000~findings, only approximately 50 require LLM analysis, at a total cost of \$0.50--2.00, compared to \$10--40 for sending all 1,000 to an LLM.

Existing tools treat vulnerability detection as a \textit{classification} problem: given a code snippet, output ``vulnerable'' or ``safe.'' This framing ignores two critical dimensions. First, it ignores \textit{cost}: a pattern match (\$0.001) and an LLM call (\$0.40) differ by three orders of magnitude. Second, it ignores \textit{confidence}: a classifier at 99\% certainty and one at 51\% certainty both output the same label, but only the second needs additional analysis. Triage asks: ``Given what we know so far, do we know enough to act, or do we need to investigate further?'' Our framework operationalizes this question at every stage of the cascade.

\subsection{The Software Vulnerability Crisis}
% -----------------------------------------------------------------------------

The NVD recorded 25,227~CVEs in 2023, up from 20,174 in 2021 and 12,174 in 2019~\cite{nvd2023}. Applications assembled from hundreds of open-source dependencies propagate single vulnerabilities to thousands of downstream consumers, as Log4Shell (CVE-2021-44228) demonstrated across an estimated 35,000~Java packages. The Edgescan 2025 report found that 45.4\% of discovered vulnerabilities in large enterprises remain unresolved at 12~months~\cite{edgescan2025}.

SAST tools are the first line of automated defense: fast, deterministic, and CI/CD-integrated. They are also overwhelmingly noisy. Ghost Security's 2025 study scanned nearly 3,000~repositories across Go, Python, and PHP and found that over 91\% of all flagged issues were false positives~\cite{ghostsecurity2025}. Table~\ref{tab:fp-rates} summarizes false positive rates on the OWASP Benchmark v1.2~\cite{siftingnoise2026}.

The downstream consequence is alert fatigue. Johnson et al.~\cite{johnson2013} found that 40--60\% of SAST warnings are dismissed without investigation. Each finding requires 10--20~minutes of expert review; an organization producing 500~findings per release cycle faces 80--160~hours of triage, 90\% of which yields no actionable result. Organizations that run weekly scans across 100+~repositories face annual triage costs exceeding \$1~million for a process that catches fewer than one in ten genuine vulnerabilities. The rational organizational response is to reduce scan frequency or abandon SAST entirely, each of which increases the window of exposure.

Figure~\ref{fig:iceberg} illustrates this dynamic. The visible portion, CVEs that are reported and patched, represents a fraction of the actual attack surface. Beneath it sit findings flagged but ignored, vulnerabilities in code paths SAST cannot reach, and patterns no existing rule covers.

Automated vulnerability detection has evolved through three generations. First-generation pattern matchers (Flawfinder, RATS) are fast but have no notion of data flow. Second-generation taint analyzers (CodeQL~\cite{codeql2024}, Fortify, Coverity) track data propagation from sources to sinks but cannot follow taint through dynamic dispatch or reflection. Third-generation learned representations (GNNs and LLMs) promised to bypass hand-written rules but collapse on realistic distributions or hallucinate at scale.

Rice's Theorem~\cite{rice1953} guarantees that any sound static analysis must over-approximate and will inevitably produce false positives. Abstract interpretation, the formal basis for sound SAST, trades precision for soundness: if the analysis cannot prove a path is safe, it assumes the path is dangerous. The question is not whether SAST produces false positives, but how to manage them. Current practice manages them by ignoring them. We propose a different approach: quantify the uncertainty of each finding and route findings whose uncertainty exceeds a threshold to the analysis modality best equipped to resolve them. A hardcoded credential does not need an LLM to confirm it. A taint path through three abstraction layers does.


% -----------------------------------------------------------------------------
\subsection{The Generalization Collapse}
% -----------------------------------------------------------------------------

Graph Neural Networks (GNNs) operating on Code Property Graphs (CPGs)~\cite{cpg2014} promised to learn vulnerability patterns directly from data. Devign~\cite{devign2019}, ReGVD~\cite{regvd2022}, and LineVul~\cite{linevul2022} reported F1 scores between 60\% and 93\% on their respective benchmarks. These numbers are unreliable. The Real-Vul study~\cite{realvul2024} re-evaluated state-of-the-art models on a realistic dataset with strict deduplication, temporal splits, and label verification. Models achieving \textbf{93\% F1 on BigVul dropped to 2\% F1 on Real-Vul}. PrimeVul~\cite{primevul2024} independently confirmed these findings: a 7B-parameter model scored 68.26\% F1 on BigVul but only \textbf{3.09\% F1 on PrimeVul}. The root causes are data leakage, unrealistic 50/50 label distributions, and temporal contamination.

LLMs face a parallel credibility problem. While they bring contextual understanding that statistical models lack (an LLM can reason about whether a parameterized query actually prevents injection), their outputs are neither deterministic nor verifiable. LLM4Vuln~\cite{llm4vuln2024} documented hallucination rates approaching 90\% when LLMs serve as vulnerability detectors: models fabricate CVE identifiers and confidently assert exploitability for provably safe code. Steenhoek et al.~\cite{steenhoek2024} found that ChatGPT produces inconsistent outputs in 47--76\% of repeated coding tasks. Even with temperature set to zero, LLMs exhibit non-determinism due to floating-point arithmetic variations and batching effects. A security tool that gives different answers on different invocations cannot serve as the sole basis for triage decisions.

The three-generation pattern is clear: each generation solves the previous generation's primary weakness while introducing a new one. The next advance will not come from a fourth generation of single-modality analysis, but from a principled method of combining the existing three so that each compensates for the others' weaknesses.

These failures motivate two design choices. First, the GNN is never a standalone oracle. Guo et al.~\cite{guocalibration2017} demonstrated that modern neural networks are systematically overconfident. Conformal prediction~\cite{angelopoulos2023, vovk2005} provides a principled alternative: rather than outputting a single label with an uncalibrated probability, it produces a \textit{prediction set} $\mathcal{C}(x)$ satisfying $P(y_{\text{true}} \in \mathcal{C}(x)) \geq 1 - \alpha$ for any user-specified $\alpha$, distribution-free. In our framework, singleton prediction sets resolve at Stage~2; ambiguous sets escalate to the LLM stage.

Second, the LLM is never a single-pass classifier. Multi-agent debate~\cite{multiagentdebate2023} improves factual accuracy by 20--40\% by forcing agents to respond to counterarguments. We apply this with a domain-specific design: an attacker agent constructs exploit paths while a defender agent identifies sanitizers, and a deterministic consensus engine reconciles their structured verdicts.

\section{\uppercase{Problem Statement and Objectives}}
% =============================================================================

SAST tools produce 68--95\% false positives, ML models collapse by 90+ percentage points on realistic data, and LLMs hallucinate at rates that preclude deterministic guarantees. No existing system combines cheap fast screening with principled uncertainty-driven escalation to address all three failure modes simultaneously. Recent hybrid systems (IRIS~\cite{iris2025}, Vulnhalla~\cite{vulnhalla2025}, ZeroFalse~\cite{zerofalse2025}, LLMxCPG~\cite{llmxcpg2025}) combine modalities but apply their expensive stages uniformly, so cost scales linearly with finding volume. At \$0.10--0.40 per LLM call and hundreds of findings per scan, organizations running nightly scans across hundreds of repositories face annual LLM costs exceeding \$500,000. An effective system must invoke expensive analysis only when cheap analysis is insufficient.

We formalize the gap as follows: \textit{How can we construct a vulnerability detection framework that allocates analysis effort in proportion to finding difficulty, provides calibrated uncertainty estimates at each stage, and resolves conflicting evidence from heterogeneous analysis modalities, while maintaining economic viability for real-world deployment?}

Consider a concrete example. A SAST tool flags a call to \texttt{subprocess.run()} with user-controlled input as a potential command injection (CWE-78). The tool cannot determine whether the input is sanitized upstream. A GNN analyzes the Code Property Graph and outputs $P(\text{vulnerable}) = 0.62$, above threshold but not highly confident. An LLM examines the code and finds that the input passes through \texttt{shlex.quote()} before reaching the shell command. Which verdict should the system trust? The answer depends on the CWE category (injection CWEs benefit from LLM context), the confidence level of each modality (the GNN is ambiguous, the LLM is specific), and the deployment cost constraints (the LLM call was justified because SAST uncertainty was high). Our framework formalizes this reasoning into a principled pipeline.

We decompose this problem into five objectives:

\begin{enumerate}
    \item \textbf{Selective escalation.} Design a 4-factor uncertainty scoring model that resolves $\geq$75\% of findings at the SAST stage (achieved: 85\%), yielding $\geq$85\% fewer LLM API calls than uniform analysis.
    \item \textbf{Calibrated graph classification.} Apply conformal prediction (APS, $\alpha{=}0.1$) to GNN outputs, providing distribution-free coverage guarantees and $\geq$60\% singleton rate.
    \item \textbf{Adversarial LLM validation.} Replace single-model LLM classification with a dual-agent attacker-defender protocol and five deterministic consensus rules.
    \item \textbf{CWE-adaptive fusion.} Fuse SAST, GNN, and LLM evidence using per-CWE weight profiles (14 profiles) configured based on domain expertise regarding each modality's empirical strengths.
    \item \textbf{End-to-end integration.} Deliver a working pipeline for five languages (Python, JavaScript, Java, C/C++, Go) with SARIF~2.1.0 output and graceful degradation when tools are unavailable.
\end{enumerate}

\section{\uppercase{Research Questions and Contributions}}
% =============================================================================

We structure our investigation around five research questions, each tied to a specific contribution. The questions form a progression: RQ1 and RQ2 evaluate the two escalation mechanisms, RQ3 evaluates the LLM stage's dual-agent design, RQ4 evaluates the fusion mechanism, and RQ5 evaluates the end-to-end system. \textbf{C1: Uncertainty-driven cascading escalation.} A 4-factor uncertainty score routes findings through a SAST $\to$ Graph $\to$ LLM cascade:
\begin{equation}
U = 0.4 \cdot U_{\text{conf}} + 0.3 \cdot U_{\text{comp}} + 0.2 \cdot U_{\text{nov}} + 0.1 \cdot U_{\text{confl}} + U_{\text{sev}}
\label{eq:ch1-uncertainty}
\end{equation}
\noindent where the four factors capture confidence (inverse of SAST tool confidence), complexity (taint path length and call depth), novelty (rarity of the CWE pattern), and conflict (disagreement between SAST rules). The severity adjustment $U_{\text{sev}}$ biases CRITICAL findings toward escalation (+0.15), reflecting the asymmetry that the cost of missing a critical vulnerability exceeds the cost of an unnecessary API call by orders of magnitude. Findings with $U < 0.5$ resolve at the SAST stage; this threshold is configurable per deployment. \textit{Result}: 85\% SAST resolution across 184~findings in 15~repositories, the first system to formalize SAST-to-LLM escalation as a decision boundary in uncertainty space.

\textbf{C2: Conformal prediction for code security.} Adaptive Prediction Sets (APS) wrap GNN outputs in distribution-free coverage guarantees:
\begin{equation}
P\!\left(y_{\text{true}} \in \mathcal{C}(x)\right) \geq 1 - \alpha
\label{eq:ch1-conformal}
\end{equation}
\noindent Singleton sets resolve at Stage~2; ambiguous sets (both labels included) escalate to Stage~3. This is, to our knowledge, the first application of conformal prediction to vulnerability detection. \textit{Result}: 69.1\% singleton rate at $\alpha = 0.1$, with 90\% coverage guarantee.

\textbf{C3: Adversarial dual-agent consensus.} An attacker agent constructs exploit paths while a defender agent identifies sanitizers; a consensus engine reconciles their structured verdicts via five deterministic rules. Both agents receive identical RAG context from 200K+ NVD entries and 900+ CWE descriptions. \textit{Result}: five decision rules with full CVSS~v3.1 scoring for every validated finding.

\textbf{C4: CWE-adaptive score fusion.} Per-CWE weight profiles reflect empirical modality strengths: injection CWEs receive LLM-heavy weights ($\alpha=0.25, \beta=0.25, \gamma=0.50$), cryptographic CWEs receive SAST-heavy weights ($\alpha=0.50, \beta=0.20, \gamma=0.30$), and memory CWEs receive graph-heavy weights ($\alpha=0.20, \beta=0.50, \gamma=0.30$). \textit{Result}: 14~weight profiles configured across CWE categories.

\textbf{C5: End-to-end cascade framework.} The integrated pipeline supports five languages, two LLM providers, SARIF~2.1.0 output, and graceful degradation when tools are unavailable. \textit{Result}: GNN F1 = 0.781 (V4 peak) / 0.750 (deployed V5 with conformal prediction), 100\% finding resolution across 184 findings, 85--95\% cost reduction versus full-LLM baseline.

The five contributions form a progression from individual components (C1--C4) to the integrated system (C5), and they are independently valuable. An organization that cannot deploy the GNN still benefits from uncertainty-driven SAST escalation (C1) and dual-agent LLM validation (C3). An organization without LLM API access still benefits from SAST pre-screening and conformal prediction (C1 + C2). The cascade degrades gracefully: when a stage is unavailable, its findings escalate to the next available stage rather than being lost. The full architecture is presented in Chapter~3, Figure~\ref{fig:full-arch}.
