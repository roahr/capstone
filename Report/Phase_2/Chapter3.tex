The literature review in Chapter~2 exposed three structural gaps in current vulnerability detection research: (1)~static analysis tools produce false-positive rates exceeding 90\% because they lack semantic context \cite{ghostsecurity2025}, (2)~graph neural networks for vulnerability detection plateau at 63--65\% accuracy when trained on single-representation inputs \cite{devign2019}, and (3)~large language models hallucinate plausible but incorrect security verdicts when operating without grounding evidence \cite{primevul2024}. Each gap corresponds to a strength of a different analysis modality, which motivated us to design a multi-stage cascade that combines all three. This chapter specifies the four modules of the \texttt{sec-c} framework, the uncertainty-driven routing logic between them, and the formal guarantees that bind the stages together. Every design choice is justified by the trade-off it resolves, and every constant is traced to its implementation in the codebase.

\subsection{Cascade Philosophy}

The central claim of this work is that vulnerability triage operates under a strict cost--accuracy trade-off: cheap analysis resolves easy cases, expensive analysis should be reserved for hard ones. We instantiate this principle as a four-stage cascade where each finding enters at the cheapest stage and escalates only when the current stage cannot resolve it with sufficient confidence.

Stage~1 (\textbf{SAST Pre-Screening}) applies Tree-sitter pattern matching and CodeQL taint analysis. Each finding receives a four-factor uncertainty score $U$; those with $U < 0.5$ are resolved immediately. Stage~2 (\textbf{Graph-Augmented Validation}) builds a Joern Code Property Graph (CPG), backward-slices it, produces GraphCodeBERT embeddings concatenated with six structural features, and classifies via MiniGINv3---a three-layer Graph Isomorphism Network. A conformal prediction layer wraps the classifier: singleton prediction sets resolve; ambiguous sets escalate. Stage~3 (\textbf{LLM Dual-Agent Validation}) dispatches an Attacker agent (red team) and a Defender agent (blue team) via Gemini~2.5, fuses their verdicts through a five-rule consensus protocol, and computes CVSS~v3.1 scores. Stage~4 (\textbf{Score Fusion and Reporting}) applies CWE-adaptive weighted fusion and emits SARIF~2.1.0 output with an interactive HTML dashboard.

\subsection{Design Principles}

Three principles guided the architecture.

\emph{Uncertainty-driven escalation} ensures that the routing decision between stages is principled rather than heuristic: findings escalate because their uncertainty exceeds a calibrated threshold, not because they match a hard-coded rule. Traditional multi-tool pipelines use severity-based routing (escalate all CRITICAL findings), which wastes expensive analysis on trivially confirmed critical findings and misses ambiguous medium-severity findings. Our four-factor uncertainty score captures aspects that severity alone cannot: taint-path complexity, CWE novelty, and inter-tool disagreement.

\emph{Graceful degradation} means the pipeline continues to operate when optional tools are unavailable---if Joern is not installed, findings bypass Stage~2 and escalate directly to Stage~3; if no LLM API key is configured, findings remain at their current resolution. This principle is essential for practical adoption: a developer running \texttt{sec-c scan} on a laptop should receive useful output in seconds, while a CI/CD pipeline with full infrastructure should receive deeper analysis.

\emph{Auditability} requires that every intermediate score, every routing decision, and every agent verdict is recorded in the \texttt{Finding} Pydantic model (defined in \texttt{src/sast/sarif/schema.py}), so the final SARIF report traces each finding's journey through the cascade. A security engineer reviewing a CONFIRMED finding can inspect the SAST confidence, the uncertainty breakdown, the conformal prediction set, the attacker's payload, the defender's sanitiser analysis, and the fused score---all from a single SARIF result object.

\section{\uppercase{Stage 1: SAST Pre-Screening}}
\label{sec:sast}
% ====================================================================

The first stage serves two purposes: it generates the initial set of candidate vulnerability findings, and it computes a principled uncertainty score that determines whether each finding can be resolved immediately or must be escalated. This is Contribution~C1: \emph{uncertainty-driven cascade routing}.

\subsection{Tree-sitter Pattern Matching}

Before invoking CodeQL (which requires database construction taking 1--5 minutes), we run a Tree-sitter pre-screening pass that completes in under 100~ms per file. The \texttt{TreeSitterScanner} parses the source into a concrete syntax tree and matches 24 vulnerability-indicative patterns across five languages (Python, JavaScript, Java, C/C++, Go). These patterns detect common anti-patterns---\texttt{eval()} on user input, \texttt{shell=True} in subprocess calls, SQL string concatenation, missing CSRF tokens---and produce findings with a baseline confidence of 0.6--0.8 (lower than CodeQL's taint-aware confidence of 0.8--0.95). Tree-sitter pre-screening ensures that even when CodeQL is unavailable, the pipeline has a non-empty set of findings to triage.

\subsection{CodeQL Taint Analysis}

For repositories where CodeQL databases can be constructed (or pre-built databases downloaded via \texttt{GITHUB\_TOKEN}), we run the \texttt{security-extended} query suite. CodeQL traces data flow from sources (user input, environment variables, file reads) through intermediate transformations to sinks (SQL queries, OS commands, HTML output), producing findings with full taint-flow paths. Each \texttt{TaintFlow} object records an ordered list of \texttt{TaintFlowStep} entries, each with a \texttt{Location} (file, line, column, snippet) and a \texttt{kind} label (\texttt{source}, \texttt{intermediate}, \texttt{sanitizer}, or \texttt{sink}). The taint-flow length and interprocedural depth feed directly into the uncertainty scorer.

\subsection{Four-Factor Uncertainty Scoring}

We claim that a single confidence score from a SAST tool is insufficient for routing decisions. A finding might have high SAST confidence yet involve a 7-hop interprocedural taint path that the tool cannot fully verify. We therefore decompose uncertainty into four independent factors, implemented in the \texttt{UncertaintyScorer} class (\texttt{src/sast/uncertainty/scorer.py}).

The composite uncertainty score is:

\begin{align}
U = w_{\text{conf}} \cdot U_{\text{conf}} + w_{\text{comp}} \cdot U_{\text{comp}} + w_{\text{nov}} \cdot U_{\text{nov}} + w_{\text{confl}} \cdot U_{\text{confl}} + \delta_{\text{sev}}
\label{eq:uncertainty}
\end{align}

\noindent where the weights are $w_{\text{conf}} = 0.4$, $w_{\text{comp}} = 0.3$, $w_{\text{nov}} = 0.2$, $w_{\text{confl}} = 0.1$, and $\delta_{\text{sev}}$ is a post-hoc severity adjustment. The result is clamped to $[0, 1]$.

\textbf{Factor 1: Confidence Uncertainty} ($U_{\text{conf}}$). The inverse of the SAST tool's self-reported confidence: $U_{\text{conf}} = 1.0 - c_{\text{sast}}$, clamped to $[0,1]$. A tool reporting 90\% confidence yields $U_{\text{conf}} = 0.1$; a tool reporting no confidence yields $U_{\text{conf}} = 1.0$. This factor receives the highest weight (0.4) because SAST confidence is the most direct signal of detection reliability.

\textbf{Factor 2: Complexity Uncertainty} ($U_{\text{comp}}$). Computed from two sub-signals averaged equally:
\[
U_{\text{comp}} = \frac{\text{hop\_score} + \text{depth\_score}}{2}
\]
where $\text{hop\_score} = \text{clamp}((l_{\text{taint}} - 1) / 4)$ with $l_{\text{taint}}$ the taint-flow length and the denominator set to $\texttt{MAX\_TAINT\_HOPS} - 1 = 4$, and $\text{depth\_score} = \text{clamp}(d_{\text{interproc}} / 5)$ with $d_{\text{interproc}}$ the interprocedural call depth. Longer taint paths and deeper call chains are harder for SAST tools to verify \cite{johnson2013}, so they increase uncertainty.

\textbf{Factor 3: Novelty Uncertainty} ($U_{\text{nov}}$). A binary classification of the CWE pattern: 25 well-studied CWEs (SQL injection, XSS, buffer overflows, path traversal, etc.) receive $U_{\text{nov}} = 0.15$, rare CWEs receive $U_{\text{nov}} = 0.85$, and findings with no CWE identifier receive $U_{\text{nov}} = 1.0$. We chose a discrete rather than continuous mapping because SAST tool accuracy drops sharply outside the most common CWE categories---there is no smooth gradient.

\textbf{Factor 4: Conflict Uncertainty} ($U_{\text{confl}}$). Measures inter-tool disagreement when multiple SAST tools flag the same location. If only one tool produced the finding, $U_{\text{confl}} = 0.0$. If multiple tools agree, $U_{\text{confl}} = 0.1$. Soft disagreement (e.g., one says ``likely'' and another says ``potential'') yields $U_{\text{confl}} = 0.5$. Hard disagreement (one says ``safe'', another says ``vulnerable'') yields $U_{\text{confl}} = 1.0$.

Algorithm~\ref{alg:uncertainty} formalises the scoring procedure.

\begin{algorithm}[H]
\caption{Four-Factor Uncertainty Scoring}
\label{alg:uncertainty}
\begin{algorithmic}[1]
\Require Finding $f$ with fields: $c_{\text{sast}}$, $l_{\text{taint}}$, $d_{\text{interproc}}$, $\text{cwe\_id}$, severity, tool\_verdicts
\Ensure $U \in [0, 1]$
\State $U_{\text{conf}} \gets \text{clamp}(1.0 - c_{\text{sast}})$
\State $\text{hop\_score} \gets \text{clamp}((l_{\text{taint}} - 1) / 4)$
\State $\text{depth\_score} \gets \text{clamp}(d_{\text{interproc}} / 5)$
\State $U_{\text{comp}} \gets (\text{hop\_score} + \text{depth\_score}) / 2$
\If{$\text{cwe\_id} \in \text{COMMON\_CWE\_IDS}$ (25 entries)}
    \State $U_{\text{nov}} \gets 0.15$
\ElsIf{$\text{cwe\_id}$ is present}
    \State $U_{\text{nov}} \gets 0.85$
\Else
    \State $U_{\text{nov}} \gets 1.0$
\EndIf
\State $U_{\text{confl}} \gets \Call{ConflictScore}{\text{tool\_verdicts}}$ \Comment{0.0/0.1/0.5/1.0}
\State $\delta_{\text{sev}} \gets \text{SEVERITY\_ADJ}[\text{severity}]$ \Comment{+0.15/+0.10/0.00/$-$0.05}
\State $U \gets \text{clamp}(0.4 \cdot U_{\text{conf}} + 0.3 \cdot U_{\text{comp}} + 0.2 \cdot U_{\text{nov}} + 0.1 \cdot U_{\text{confl}} + \delta_{\text{sev}})$
\State \Return $U$
\end{algorithmic}
\end{algorithm}

\subsection{Escalation Routing}

The \texttt{EscalationRouter} class (\texttt{src/sast/router.py}) applies four rules to determine whether a finding escalates from Stage~1 to Stage~2. A finding escalates if \emph{any} rule fires. Table~\ref{tab:routing-rules} enumerates these rules.

\begin{table}[H]
\centering
\caption{SAST-to-Graph escalation rules. A finding escalates if any rule fires.}
\label{tab:routing-rules}
\begin{tabular}{clp{7cm}}
\toprule
\textbf{Rule} & \textbf{Condition} & \textbf{Rationale} \\
\midrule
R1 & $U \geq 0.5$ & High composite uncertainty; SAST alone is unreliable \\
R2 & $l_{\text{taint}} > 3$ & Long taint paths exceed SAST verification capacity \\
R3 & \texttt{is\_interprocedural} = True & Cross-file flows require structural analysis \\
R4 & Interprocedural AND \texttt{CRITICAL} & Critical cross-file findings always escalate regardless of $U$ \\

\end{tabular}

Findings that do not trigger any rule are resolved at Stage~1 with verdict \texttt{SAFE} and \texttt{stage\_resolved = SAST}. In evaluation across 15 repositories, 85\% of findings resolved at Stage~1, consistent with the observation that the majority of SAST findings are either true positives with high confidence or obvious false positives \cite{johnson2013}.

For the Graph-to-LLM transition, the router applies a single rule: if the conformal prediction set is ambiguous (contains both ``safe'' and ``vulnerable''), the finding escalates. Singleton prediction sets (either \{``safe''\} or \{``vulnerable''\}) resolve at Stage~2 with the corresponding verdict.

\section{\uppercase{Stage 2: Graph-Augmented Validation}}
\label{sec:graph}
% ====================================================================

Stage~2 addresses a key limitation of SAST tools: they produce taint paths but cannot reason about the structural context surrounding those paths. A missing bounds check, an uncalled sanitiser, a dead-code branch---these structural properties require graph-level analysis. This section describes how we construct Code Property Graphs, slice them to reduce noise, extract features, classify with a graph neural network, and wrap the classifier in a conformal prediction layer that provides coverage guarantees. This constitutes Contribution~C2: \emph{conformal prediction for principled escalation}.

\subsection{Code Property Graph Construction}

We use Joern \cite{cpg2014} to generate Code Property Graphs that unify four program representations. The \texttt{JoernCPGBuilder} class (\texttt{src/graph/cpg/builder.py}) invokes \texttt{joern-parse} to produce a CPG binary and \texttt{joern-export} to convert it to GraphML, which is then parsed into a NetworkX \texttt{DiGraph}. Table~\ref{tab:cpg-types} enumerates the 14 node types and 6 edge types.

\subsection{Backward Slicing}

Full CPGs for real-world codebases contain thousands of nodes, most of which are irrelevant to a specific vulnerability. We apply backward slicing \cite{cpg2014} from the finding's sink to extract only the security-relevant subgraph. The \texttt{BackwardSlicer} class (\texttt{src/graph/slicing/slicer.py}) performs BFS backward from the sink node along DDG, REACHING\_DEF, and CDG edges (and optionally CALL edges for inter-procedural analysis) up to a maximum depth of 10.

After slicing, the algorithm expands the result by $\pm 5$ lines of AST context around each slice node, ensuring the extracted code remains readable for subsequent LLM analysis. Typical graph reduction is 67--91\%, consistent with LLMxCPG~\cite{llmxcpg2025}: a CPG with 2,000 nodes is sliced to 180--660 nodes, dramatically reducing the input size for the GNN and any downstream LLM prompts.

We chose backward slicing over forward slicing because vulnerability detection is sink-driven: the security question is ``can untrusted data reach this dangerous operation?'', which requires tracing backward from the sink to potential sources. Forward slicing from a source would include all code that the source data \emph{might} affect, much of which is irrelevant to the specific sink under investigation.

The slicer also locates the taint-flow source node (if identifiable from the finding's \texttt{TaintFlow}) and performs a shallow backward slice (depth~3) from the source, merging both slices. This captures the source's input-validation context without exploding the graph size. The merged slice is stored as a new \texttt{nx.DiGraph} with graph-level metadata (\texttt{sliced=True}, \texttt{finding\_id}) for downstream traceability.

\subsection{Feature Extraction}

Each node in the sliced CPG receives a 774-dimensional feature vector composed of two parts: 768 dimensions from GraphCodeBERT \cite{graphcodebert2021} and 6 hand-crafted structural features.

\textbf{GraphCodeBERT Embeddings (768 dims).} The \texttt{CodeEmbedder} class (\texttt{src/graph/features/embeddings.py}) loads \texttt{microsoft/graphcodebert-base} (a RoBERTa-based model pre-trained on code and data-flow graphs) and extracts the CLS token embedding for each node's code text. Embeddings are computed in batches of 32 and cached in an LRU cache (capacity 4,096) to avoid redundant computation. We chose GraphCodeBERT over CodeBERT \cite{codebert2020} because its data-flow-aware pre-training produces representations that better capture taint-relevant semantics.

\textbf{Structural Features (6 dims).} The \texttt{NodeFeatureExtractor} class (\texttt{src/graph/features/node\_features.py}) computes six features per node, all normalised to $[0, 1]$. Table~\ref{tab:structural-features} details each feature.

The \texttt{is\_sink} and \texttt{is\_source} features encode domain knowledge about taint-relevant API functions (e.g., \texttt{execute}, \texttt{eval}, \texttt{pickle.loads} for sinks; \texttt{request}, \texttt{argv}, \texttt{environ} for sources) without requiring full taint analysis. The \texttt{language\_id} feature was introduced in V3 (replacing the 5-feature V2 configuration) to allow the model to learn language-specific vulnerability patterns.

The final node feature vector is $\mathbf{x}_v = [\mathbf{e}_v \| \mathbf{s}_v] \in \mathbb{R}^{774}$, where $\mathbf{e}_v \in \mathbb{R}^{768}$ is the GraphCodeBERT embedding and $\mathbf{s}_v \in \mathbb{R}^{6}$ is the structural feature vector.

\subsection{MiniGINv3 Architecture}

We chose the Graph Isomorphism Network (GIN) \cite{xu2019gin} over the Graph Attention Network (GAT) based on a principled expressiveness argument. Xu et al.\ (2019) proved that GNNs using mean or max aggregation (as GAT does) are strictly less expressive than the Weisfeiler-Leman (1-WL) graph isomorphism test, while sum aggregation (as GIN uses) is \emph{as powerful as} 1-WL. For vulnerability detection, code graphs often differ in subtle structural ways---a missing bounds check removes one edge, a dead-code branch adds nodes that never execute. GAT's weighted-mean aggregation can conflate these structurally distinct neighbourhoods, while GIN's injective sum aggregation preserves the differences.

Table~\ref{tab:gat-vs-gin} compares the two architectures.

The MiniGINv3 architecture processes the 774-dimensional input through a projection layer, three GIN convolutional layers with residual connections, dual global pooling, a classification head, and a confidence head. Table~\ref{tab:miniginv3-layers} gives the layer-by-layer specification.

\bottomrule
\end{tabular}

Each GIN layer applies the update rule:

\begin{align}
\mathbf{h}_v^{(k)} = \text{MLP}^{(k)}\!\left((1 + \epsilon^{(k)}) \cdot \mathbf{h}_v^{(k-1)} + \sum_{u \in \mathcal{N}(v)} \mathbf{h}_u^{(k-1)}\right)
\label{eq:gin-update}
\end{align}

\noindent where $\epsilon^{(k)}$ is a learnable parameter (\texttt{train\_eps=True}), $\mathcal{N}(v)$ is the set of neighbours of node $v$, and $\text{MLP}^{(k)}$ is a two-layer perceptron (384$\to$768$\to$384) with BatchNorm and ReLU. After each GIN layer, a residual connection adds the input: $\mathbf{h}_v^{(k)} \gets \mathbf{h}_v^{(k)} + \mathbf{h}_v^{(k-1)}$.

The dual pooling concatenates global mean pooling and global add pooling, yielding a 768-dimensional graph-level embedding. We use dual pooling because mean pooling captures the ``average node behaviour'' while add pooling preserves the total magnitude of node features---a property that matters when the number of vulnerable nodes varies.

The confidence head is trained with an auxiliary binary cross-entropy loss:
\[
\mathcal{L}_{\text{conf}} = 0.1 \times \text{BCE}(\hat{c}, \mathbb{1}[\arg\max(\hat{y}) = y])
\]
where $\hat{c}$ is the predicted confidence and $\mathbb{1}[\cdot]$ indicates whether the classifier's prediction is correct. This trains the confidence head to estimate prediction reliability, providing a secondary uncertainty signal independent of the softmax output.

\subsection{GNN Training Configuration}

Training uses the AdamW optimiser with a learning rate of $3 \times 10^{-4}$, weight decay of $10^{-3}$, and gradient clipping at 0.5. The learning rate schedule consists of a 5-epoch linear warmup followed by cosine decay. We train with a batch size of 64 and dropout rate of 0.35 (reduced from 0.4 in V3 to V5 based on validation performance). Class imbalance is handled through a weighted cross-entropy loss with a vulnerability class weight of 1.5. Early stopping with a patience of 20 epochs prevents overfitting.

The total loss combines classification and confidence objectives:
\[
\mathcal{L}_{\text{total}} = \mathcal{L}_{\text{CE}}(\hat{y}, y; w_{\text{vuln}}=1.5) + 0.1 \times \mathcal{L}_{\text{conf}}
\]

We train on a composite dataset drawn from Juliet Test Suite \cite{juliet2017}, CVEfixes \cite{cvefixes2021}, and DiverseVul \cite{diversevul2023}. The dataset is split 60/15/15/10 for training, validation, calibration, and test (the calibration set is reserved exclusively for conformal prediction and the test set for final evaluation; neither is used during training or hyperparameter selection). Graph construction constraints limit each sample to a maximum of 300 nodes (larger graphs are truncated), with a minimum code length of 20 characters and 3 lines to filter trivial samples.

The evolution from V3 to the deployed V5 involved two changes: dropout was reduced from 0.4 to 0.35 (V4), and label smoothing was removed entirely (V5, from 0.1 in V3/V4). We removed label smoothing because it interfered with conformal calibration---smoothed targets shift the logit distribution in ways that increase the calibration set's nonconformity scores, leading to wider (more ambiguous) prediction sets. Without label smoothing, the deployed model produces tighter prediction sets and fewer unnecessary escalations to Stage~3.

\subsection{Conformal Prediction Layer}

A softmax output of 0.7 does not mean the model is 70\% correct---modern neural networks are poorly calibrated \cite{guocalibration2017}. We need a mechanism that converts the GNN's output into a prediction set with a formal coverage guarantee. This is where conformal prediction enters: it provides distribution-free, finite-sample coverage guarantees without assumptions about the data distribution \cite{angelopoulos2023, vovk2005}.

We implement Adaptive Prediction Sets (APS) \cite{romano2020} in the \texttt{ConformalPredictor} class (\texttt{src/graph/uncertainty/conformal.py}), configured with $\alpha = 0.1$ (90\% coverage guarantee).

\textbf{Conformal Temperature Scaling (ConfTS).} Before computing softmax probabilities, we apply temperature scaling with $T = 0.95$ (deployed value, overriding the default $T=0.2$ in \texttt{configs/default.yaml} based on the grid search described in Chapter~4). The temperature-scaled softmax is:

\begin{align}
\hat{\pi}_i = \frac{\exp(z_i / T)}{\sum_j \exp(z_j / T)}
\label{eq:confts}
\end{align}

\noindent where $z_i$ are the raw logits. Temperature scaling improves calibration without changing the rank order of predictions \cite{guocalibration2017}, which in turn tightens the conformal prediction sets (fewer ambiguous sets means fewer unnecessary escalations).

\textbf{Calibration Phase.} On a held-out calibration set disjoint from both training and test data, we compute nonconformity scores for each sample $i$ with true label $y_i$:

\begin{enumerate}
    \item Sort classes by descending softmax probability: $\hat{\pi}_{(1)} \geq \hat{\pi}_{(2)} \geq \ldots$
    \item Compute cumulative sums: $C_j = \sum_{k=1}^{j} \hat{\pi}_{(k)}$
    \item The nonconformity score is $s_i = C_{\text{rank}(y_i)}$, the cumulative sum at the position where the true label first appears.
\end{enumerate}

The quantile threshold is then:

\begin{align}
\hat{q} = \text{Quantile}\!\left(\{s_1, \ldots, s_n\},\; \frac{\lceil (n+1)(1-\alpha) \rceil}{n}\right)
\label{eq:aps-quantile}
\end{align}

\noindent using the ``higher'' interpolation method, clamped to $[0, 1]$.

\textbf{Coverage Guarantee.} For any new test sample $(X, Y)$ exchangeable with the calibration data, APS guarantees:

\begin{align}
P(Y \in \mathcal{C}(X)) \geq 1 - \alpha
\label{eq:coverage}
\end{align}

This guarantee holds regardless of the model's accuracy---it is a property of the conformal procedure, not the classifier. The only assumption is exchangeability between calibration and test samples \cite{barber2023}, which is satisfied when both sets are drawn i.i.d.\ from the same distribution. Importantly, the guarantee is marginal (over the randomness of the calibration set and new sample), not conditional on a specific input, but this is sufficient for our use case: we need aggregate reliability, not per-finding guarantees.

\textbf{Inference.} For a new graph, we compute temperature-scaled softmax probabilities, sort classes in descending order, and include classes greedily until the cumulative sum meets or exceeds $\hat{q}$. The possible outcomes for our binary classifier are:

\begin{itemize}
    \item \textbf{Singleton set} $\{\text{``safe''}\}$ or $\{\text{``vulnerable''}\}$: the model is confident. The finding is resolved at Stage~2.
    \item \textbf{Ambiguous set} $\{\text{``safe''}, \text{``vulnerable''}\}$: the model cannot distinguish the classes. The finding escalates to Stage~3.
\end{itemize}

Algorithm~\ref{alg:aps} formalises both phases.

\begin{algorithm}[H]
\caption{APS Calibration and Inference}
\label{alg:aps}
\begin{algorithmic}[1]
\Statex \textbf{Calibration:}
\Require Calibration set $\{(\mathbf{x}_i, y_i)\}_{i=1}^{n}$, significance level $\alpha = 0.1$, temperature $T$
\For{$i = 1, \ldots, n$}
    \State $\hat{\boldsymbol{\pi}}_i \gets \text{softmax}(\mathbf{z}_i / T)$ \Comment{Temperature-scaled probabilities}
    \State Sort classes by descending $\hat{\pi}_{i,j}$
    \State $s_i \gets \text{CumulativeSum at rank of } y_i$
\EndFor
\State $\hat{q} \gets \text{Quantile}(\{s_1, \ldots, s_n\}, \lceil(n+1)(1-\alpha)\rceil / n, \text{method=``higher''})$

\Statex
\Statex \textbf{Inference:}
\Require New sample $\mathbf{x}$, threshold $\hat{q}$
\State $\hat{\boldsymbol{\pi}} \gets \text{softmax}(\mathbf{z} / T)$
\State Sort classes by descending $\hat{\pi}_j$
\State $\mathcal{C} \gets \emptyset$
\For{$j = 1, 2, \ldots$ in sorted order}
    \State $\mathcal{C} \gets \mathcal{C} \cup \{\text{class}_j\}$
    \If{$\sum_{k \in \mathcal{C}} \hat{\pi}_k \geq \hat{q}$}
        \State \textbf{break}
    \EndIf
\EndFor
\State \Return $\mathcal{C}$, $1 - \alpha$
\end{algorithmic}
\end{algorithm}

The conformal prediction layer transforms a GNN score into a routing decision with a formal guarantee: at least 90\% of the time, the true label is in the prediction set. When the set is a singleton, we are confident; when it is not, we have principled evidence that the finding requires deeper analysis.

\section{\uppercase{Stage 3: LLM Dual-Agent Validation}}
\label{sec:llm}

Stage~3 applies large language model reasoning to the hardest cases---those that survived both SAST and graph analysis without resolution. The core insight is that a single LLM acting as a judge produces unreliable security verdicts (3.09\% F1 on realistic data \cite{primevul2024}), but an adversarial protocol with opposing agents achieves higher accuracy through structured disagreement \cite{multiagentdebate2023}. This section describes the dual-agent architecture, the consensus protocol (Contribution~C3), the CVSS scoring integration, and the RAG knowledge base.

\subsection{Dual-Agent Architecture}

The \texttt{ConsensusEngine} class (\texttt{src/llm/consensus/engine.py}) creates two specialised agents that analyse the same finding from opposing perspectives:

\textbf{Attacker Agent (Red Team).} The \texttt{AttackerAgent} attempts to construct a working exploit. Its output is an \texttt{AttackerVerdict} containing: (1)~whether the vulnerability is exploitable (\texttt{bool}), (2)~a concrete payload (\texttt{str}), (3)~attack vector, attack complexity, privileges required, and user interaction (CVSS exploitability sub-metrics), (4)~blocking factors that prevented exploitation, and (5)~a confidence score and reasoning narrative.

\textbf{Defender Agent (Blue Team).} The \texttt{DefenderAgent} searches for existing protections. Its output is a \texttt{DefenderVerdict} containing: (1)~sanitisers found (function name, type, effectiveness), (2)~access controls, (3)~framework protections (e.g., Django ORM, prepared statements), (4)~whether the taint path is feasible (\texttt{bool}), (5)~scope, confidentiality/integrity/availability impact (CVSS impact sub-metrics), and (6)~a defense coverage score $d_{\text{cov}} \in [0, 1]$ and reasoning.

Both agents extend \texttt{BaseSecurityAgent} (\texttt{src/llm/agents/base.py}), which handles Jinja2 template rendering, prompt tier selection, and response parsing. We use Gemini~2.5 Flash as the primary model (15 RPM, 500 RPD free tier) and Gemini~2.5 Pro as fallback for complex cases only. All LLM responses are requested in JSON mode (\texttt{json\_mode=True}) with a temperature of 0.1 to minimise stochastic variation across runs.

The adversarial framing is deliberate: prior work on multi-agent debate \cite{divergentthinking2023} showed that structurally opposed agents produce more factually grounded outputs than a single agent asked to ``be thorough.'' In our domain, the attacker agent is primed with an offensive mindset (``find a way to exploit this''), while the defender is primed with a defensive mindset (``find protections that prevent exploitation''). This opposition forces each agent to address the other's likely arguments, producing more complete analysis than either perspective alone.

\subsection{Consensus Protocol}

The consensus engine applies five decision rules to the combined attacker and defender verdicts. Let $e$ denote the attacker's \texttt{exploitable} flag, $c_{\text{atk}}$ the attacker's confidence, $d_{\text{cov}}$ the defender's \texttt{defense\_coverage\_score}, and $p_{\text{feas}}$ the defender's \texttt{path\_feasible} flag. Table~\ref{tab:consensus-rules} specifies each rule, and Algorithm~\ref{alg:consensus} formalises the protocol.

\begin{table}[H]
\centering
\caption{Consensus decision rules with configurable thresholds (from \texttt{engine.py}).}
\label{tab:consensus-rules}
\begin{tabular}{clll}
\toprule
\textbf{Rule} & \textbf{Condition} & \textbf{Verdict} & \textbf{Confidence} \\
\midrule
R1  & $e = \text{True} \wedge d_{\text{cov}} < 0.5$ & CONFIRMED & $\max(c_{\text{atk}}, 1 - d_{\text{cov}})$ \\
R2  & $e = \text{False} \wedge d_{\text{cov}} > 0.7$ & SAFE & $\max(d_{\text{cov}}, 1 - c_{\text{atk}})$ \\
R2b & $e = \text{False} \wedge p_{\text{feas}} = \text{False}$ & SAFE & $0.8$ (fixed) \\
R3  & $e = \text{True} \wedge d_{\text{cov}} \geq 0.5$ & LIKELY & $0.5 + 0.3(c_{\text{atk}} - d_{\text{cov}})$, clamped $[0.3, 0.85]$ \\
R4  & $e = \text{False} \wedge d_{\text{cov}} \leq 0.7$ & POTENTIAL & $0.4 + 0.2(1 - d_{\text{cov}})$, clamped $[0.2, 0.6]$ \\
\bottomrule
\end{tabular}
\end{table}

The thresholds ($0.5$ for confirmed, $0.7$ for safe, $0.8$ for infeasible confidence) are read from \texttt{configs/default.yaml} under \texttt{llm.consensus}, making them tunable without code changes.

\begin{algorithm}[H]
\caption{Dual-Agent Consensus Protocol}
\label{alg:consensus}
\begin{algorithmic}[1]
\Require AttackerVerdict $(e, c_{\text{atk}}, \text{payload}, \text{AV}, \text{AC}, \text{PR}, \text{UI})$
\Require DefenderVerdict $(d_{\text{cov}}, p_{\text{feas}}, \text{sanitisers}, \text{S}, \text{C}, \text{I}, \text{A})$
\Require Thresholds: $\tau_{\text{conf}} = 0.5$, $\tau_{\text{safe}} = 0.7$, $c_{\text{infeas}} = 0.8$
\If{$e$ \textbf{and} $d_{\text{cov}} < \tau_{\text{conf}}$} \Comment{Rule 1: Strong vulnerability evidence}
    \State \Return (CONFIRMED, $\max(c_{\text{atk}}, 1 - d_{\text{cov}})$)
\ElsIf{$\neg e$ \textbf{and} $d_{\text{cov}} > \tau_{\text{safe}}$} \Comment{Rule 2: Strong safety evidence}
    \State \Return (SAFE, $\max(d_{\text{cov}}, 1 - c_{\text{atk}})$)
\ElsIf{$\neg e$ \textbf{and} $\neg p_{\text{feas}}$} \Comment{Rule 2b: Path infeasible}
    \State \Return (SAFE, $c_{\text{infeas}}$)
\ElsIf{$e$ \textbf{and} $d_{\text{cov}} \geq \tau_{\text{conf}}$} \Comment{Rule 3: Exploitable but defended}
    \State $c \gets \text{clamp}(0.5 + 0.3(c_{\text{atk}} - d_{\text{cov}}),\; 0.3,\; 0.85)$
    \State \Return (LIKELY, $c$)
\Else \Comment{Rule 4: Not exploitable, weak defence}
    \State $c \gets \text{clamp}(0.4 + 0.2(1 - d_{\text{cov}}),\; 0.2,\; 0.6)$
    \State \Return (POTENTIAL, $c$)
\EndIf
\end{algorithmic}
\end{algorithm}

The consensus protocol resolves a fundamental tension: the attacker agent is incentivised to find exploits (high recall), while the defender agent is incentivised to find protections (high precision). By requiring both perspectives to agree before confirming or dismissing a finding, we reduce both false positives and false negatives compared to a single-agent approach.

\subsection{CVSS v3.1 Scoring}

The \texttt{compute\_cvss\_base\_score} function (\texttt{src/llm/consensus/cvss.py}) computes CVSS~v3.1 base scores from the sub-metrics produced by the attacker (AV, AC, PR, UI) and defender (S, C, I, A) agents. The computation follows the NIST specification \cite{cvss31}:

\begin{align}
\text{ISS} &= 1 - (1 - C)(1 - I)(1 - A) \label{eq:iss} \\
\text{Impact}_{\text{unchanged}} &= 6.42 \times \text{ISS} \notag \\
\text{Impact}_{\text{changed}} &= 7.52(\text{ISS} - 0.029) - 3.25(\text{ISS} - 0.02)^{15} \notag \\
\text{Exploitability} &= 8.22 \times AV \times AC \times PR \times UI \label{eq:exploitability} \\
\text{Base Score} &= \begin{cases}
    \text{roundup}(\min(\text{Impact} + \text{Exploitability},\; 10.0)) & \text{if } S = \text{unchanged} \\
    \text{roundup}(\min(1.08 \times (\text{Impact} + \text{Exploitability}),\; 10.0)) & \text{if } S = \text{changed}
\end{cases} \label{eq:cvss-base}
\end{align}

\noindent where $C$, $I$, $A$ are the confidentiality, integrity, and availability impact values $\in \{0.0, 0.22, 0.56\}$ corresponding to \{none, low, high\}, and $AV$, $AC$, $PR$, $UI$ are the exploitability metric constants defined in the CVSS~v3.1 specification.

\subsection{RAG Knowledge Base}

The \texttt{KnowledgeBase} class (\texttt{src/llm/rag/knowledge\_base.py}) provides contextual vulnerability knowledge to both agents through a hybrid retrieval architecture. We chose hybrid retrieval because semantic search alone misses keyword-specific patterns (e.g., specific CVE IDs), while keyword search alone misses semantically related but lexically different descriptions.

The knowledge base indexes two data sources: over 200,000 NVD CVE entries (parsed from NIST JSON feeds) and over 900 CWE entries (parsed from MITRE XML catalog). Retrieval combines two strategies:

\textbf{FAISS semantic search} uses \texttt{all-MiniLM-L6-v2} sentence-transformer embeddings in a flat inner-product index. The query (CWE description + code snippet) is embedded and compared against all indexed documents.

\textbf{BM25 keyword search} uses Okapi BM25 over tokenised document texts, capturing exact-match patterns that dense embeddings may miss (CVE IDs, function names, specific error messages).

Results are fused using Reciprocal Rank Fusion (RRF):

\begin{align}
\text{RRF}(d) = \sum_{r \in \{r_{\text{FAISS}}, r_{\text{BM25}}\}} \frac{w_r}{k + \text{rank}_r(d)}
\label{eq:rrf}
\end{align}

\noindent where $w_{\text{FAISS}} = 0.6$, $w_{\text{BM25}} = 0.4$, $k = 60$, and $\text{rank}_r(d)$ is the rank of document $d$ in retrieval method $r$. The top-$k = 5$ documents after RRF fusion are injected into the prompt.

\subsection{Batch Processing}

To conserve API quota under free-tier rate limits (15~RPM, 500~RPD for Flash), the consensus engine supports batch validation: up to 5 findings are analysed in a single LLM call using a structured JSON prompt. The \texttt{validate\_batch} method chunks findings into groups of \texttt{max\_batch\_size} = 5, constructs a multi-finding prompt, and parses the response into per-finding attacker and defender verdicts. If batch parsing fails, the engine falls back to individual \texttt{validate} calls. This batch strategy reduces API calls by up to 5x while maintaining the same dual-agent protocol.

\section{\uppercase{Stage 4: Score Fusion and Classification}}
\label{sec:fusion}
% ====================================================================

Stage~4 combines confidence scores from all stages that a finding traversed into a unified vulnerability score. The key design choice is CWE-adaptive weighting: different vulnerability categories benefit from different stage contributions. This is Contribution~C4.

\subsection{CWE-Adaptive Fusion Formula}

The \texttt{ScoreFusionEngine} class (\texttt{src/orchestrator/fusion.py}) computes:

\begin{align}
S_{\text{fused}} = \alpha \cdot S_{\text{SAST}} + \beta \cdot S_{\text{Graph}} + \gamma \cdot S_{\text{LLM}}
\label{eq:fusion}
\end{align}

\noindent where $(\alpha, \beta, \gamma)$ are CWE-specific weights loaded from \texttt{configs/cwe\_weights.yaml}. When a finding did not pass through a given stage (e.g., resolved at SAST, so $S_{\text{Graph}}$ and $S_{\text{LLM}}$ are absent), the engine renormalises the weights of the stages that did execute to sum to~1.0. Concretely, if only SAST executed, $S_{\text{fused}} = S_{\text{SAST}}$. If SAST and Graph executed but LLM did not, $S_{\text{fused}} = (\alpha \cdot S_{\text{SAST}} + \beta \cdot S_{\text{Graph}}) / (\alpha + \beta)$. This renormalisation ensures that findings resolved at earlier stages are not penalised for missing later-stage scores.

The component scores are extracted as follows: $S_{\text{SAST}}$ is the \texttt{sast\_confidence} field (SAST tool's self-reported confidence), $S_{\text{Graph}}$ is the \texttt{structural\_risk\_score} from \texttt{GraphValidation}, and $S_{\text{LLM}}$ is the \texttt{consensus\_confidence} from \texttt{LLMValidation}.

Table~\ref{tab:cwe-weights} shows the full set of CWE-adaptive weights. The rationale for each category follows from the strengths and weaknesses identified in Chapter~2.

\textbf{Injection CWEs} receive LLM-heavy weights because LLMs can reason about context---whether a parameterised query was used, whether the sanitiser matches the injection type, whether the framework provides automatic escaping. SAST finds the taint path but cannot evaluate the semantic adequacy of a sanitiser.

\textbf{Cryptographic CWEs} receive SAST-heavy weights because LLMs consistently fail to identify weak algorithms (they treat SHA-1 as acceptable in many contexts). SAST tools have hard-coded rules for deprecated algorithms and can reliably flag \texttt{MD5}, \texttt{DES}, and \texttt{SHA-1} usage.

\textbf{Memory safety CWEs} receive graph-heavy weights because use-after-free and null dereference vulnerabilities depend on control flow structure---which execution paths reach the dangerous access, whether a check guards the dereference. The GNN captures these structural patterns through the CPG.

\textbf{Authentication CWEs} receive the highest LLM weights ($\gamma = 0.55$) because authentication logic is inherently semantic: whether a session token is validated, whether a role check precedes a privileged operation, whether a password comparison uses constant-time comparison. SAST tools can flag missing authentication decorators but cannot evaluate whether the authentication logic is correct. The LLM agents can reason about authentication flows and identify subtle bypass conditions.

The initial weights are based on domain expertise and published benchmarks. We provide a calibration script (\texttt{scripts/calibrate\_weights.py}) that tunes these weights on labeled validation data, but the expert-derived weights serve as a strong baseline that requires minimal tuning in practice.

\subsection{Three-Tier Classification}

After computing $S_{\text{fused}}$, each finding is classified into one of three tiers. Table~\ref{tab:classification} gives the thresholds.

\begin{table}[H]
\centering
\caption{Classification thresholds applied to the fused score.}
\label{tab:classification}
\begin{tabular}{lcl}
\toprule
\textbf{Verdict} & \textbf{Threshold} & \textbf{Interpretation} \\
\midrule
CONFIRMED  & $S_{\text{fused}} \geq 0.85$ & High-confidence true positive \\
LIKELY     & $0.50 \leq S_{\text{fused}} < 0.85$ & Probable vulnerability \\
POTENTIAL  & $0.0 < S_{\text{fused}} < 0.50$ & Low confidence, needs review \\
SAFE       & (via LLM override) & False positive filtered \\
\bottomrule
\end{tabular}
\end{table}

The fusion engine also implements two LLM override rules that supersede the threshold classification: (1)~if the attacker agent confirmed exploitability \emph{and} the defender's defense coverage is below 0.3, the verdict is forced to CONFIRMED regardless of the fused score; (2)~if the attacker agent did not find an exploit \emph{and} the defender's defense coverage exceeds 0.8, the verdict is forced to SAFE. These overrides handle cases where the fused score is misleading due to a single anomalous stage score.

Each vulnerability finding in the \texttt{sec-c} framework follows a defined lifecycle from creation to final report. This lifecycle is encoded in the \texttt{Finding} Pydantic model, which serves as the universal data container across all four stages.

\textbf{State 1: NEW.} Created by Tree-sitter or CodeQL with initial fields: \texttt{rule\_id}, \texttt{cwe\_id}, \texttt{severity}, \texttt{location}, \texttt{taint\_flow}, \texttt{sast\_confidence}.

\textbf{State 2: SAST Scored.} The \texttt{UncertaintyScorer} computes and attaches the \texttt{UncertaintyScore} object (four factors, weights, severity adjustment, total).

\textbf{State 3a: SAST Resolved.} If $U < 0.5$ and no routing rules fire, the finding receives \texttt{stage\_resolved = SAST} and \texttt{verdict = SAFE}.

\textbf{State 3b: Graph Validated.} The finding receives a \texttt{GraphValidation} object containing \texttt{structural\_risk\_score}, \texttt{conformal\_prediction\_set}, \texttt{conformal\_coverage}, \texttt{taint\_path\_length}, and other structural metrics.

\textbf{State 4a: Graph Resolved.} Singleton prediction set: \texttt{stage\_resolved = GRAPH}, verdict set to \texttt{LIKELY} (if ``vulnerable'') or \texttt{SAFE} (if ``safe'').

\textbf{State 4b: LLM Validated.} The finding receives an \texttt{LLMValidation} object containing \texttt{AttackerVerdict}, \texttt{DefenderVerdict}, \texttt{consensus\_verdict}, \texttt{consensus\_confidence}, \texttt{cvss\_base\_score}, \texttt{cvss\_vector}, \texttt{cvss\_severity}, and \texttt{evidence\_narrative}.

\textbf{State 5: Score Fused.} The \texttt{ScoreFusionEngine} computes \texttt{fused\_score} and sets the final \texttt{verdict} and \texttt{nl\_explanation}.

Every intermediate value is preserved on the \texttt{Finding} object, making the entire analysis chain auditable in the final report.

\subsection{SARIF 2.1.0 Output}

The framework emits findings in SARIF~2.1.0 format (Static Analysis Results Interchange Format), the OASIS standard for static analysis output. Each finding maps to a SARIF \texttt{result} object with:

\begin{itemize}
    \item \texttt{ruleId}: the SAST rule that generated the finding
    \item \texttt{level}: mapped from the \texttt{Severity} enum (error/warning/note)
    \item \texttt{message.text}: the SAST message enriched with the verdict
    \item \texttt{locations}: file, line, column from the \texttt{Location} model
    \item \texttt{codeFlows}: full taint path from the \texttt{TaintFlow} model
    \item Custom \texttt{sec-c/*} properties: uncertainty score, stage resolved, fused score, CVSS vector, attacker/defender reasoning
\end{itemize}

\subsection{HTML Dashboard}

The \texttt{sec-c scan --dashboard} command generates an interactive HTML report with:

\begin{itemize}
    \item \textbf{Cascade funnel}: visual showing how many findings were resolved at each stage
    \item \textbf{Finding cards}: expandable per-finding detail including taint flow, graph metrics, and LLM reasoning
    \item \textbf{Severity distribution}: breakdown by CRITICAL/HIGH/MEDIUM/LOW
    \item \textbf{CWE treemap}: findings grouped by CWE category
    \item \textbf{CVSS histogram}: distribution of CVSS base scores
\end{itemize}

The dashboard is self-contained (single HTML file with embedded CSS and JavaScript) and opens automatically in the default browser.

\subsection{Graceful Degradation}

The pipeline adapts to missing tools at runtime. Table~\ref{tab:stages} shows that each stage is optional except SAST. When a stage is unavailable, its findings are either resolved with reduced confidence or escalated to the next available stage:

\begin{itemize}
    \item \textbf{CodeQL unavailable}: Tree-sitter findings are used alone (lower baseline confidence of 0.6--0.8 vs.\ CodeQL's 0.8--0.95).
    \item \textbf{Joern unavailable}: Stage~2 is skipped entirely; findings escalate directly from Stage~1 to Stage~3 (if an LLM API key is configured) or remain at Stage~1 resolution.
    \item \textbf{LLM API key missing}: Stage~3 is skipped; findings are resolved at their current stage or marked as UNRESOLVED.
    \item \textbf{GNN model missing}: Graph features are extracted but classification falls back to heuristic scoring based on structural features alone.
\end{itemize}

This degradation strategy ensures that the framework produces actionable output in any environment, from a developer laptop with no external tools to a CI/CD pipeline with full infrastructure.

\subsection{The Finding Data Model}

The unified \texttt{Finding} Pydantic model (\texttt{src/sast/sarif/schema.py}) serves as the backbone of the entire pipeline. Every stage reads from and writes to the same object, so no information is lost in translation between stages. The model contains five groups of fields:

\begin{itemize}
    \item \textbf{Identity}: \texttt{id}, \texttt{rule\_id}, \texttt{cwe\_id}, \texttt{cwe\_name}, \texttt{severity}, \texttt{language}
    \item \textbf{Location}: \texttt{Location} (file, line, column, snippet), optional \texttt{TaintFlow} (ordered list of \texttt{TaintFlowStep} with source/intermediate/sanitiser/sink labels)
    \item \textbf{Stage 1 results}: \texttt{sast\_confidence}, \texttt{sast\_message}, \texttt{sast\_tool}, \texttt{UncertaintyScore} (four factors, weights, severity adjustment, total)
    \item \textbf{Stage 2 results}: \texttt{GraphValidation} (\texttt{structural\_risk\_score}, \texttt{conformal\_prediction\_set}, \texttt{conformal\_coverage}, \texttt{taint\_path\_length}, \texttt{control\_flow\_complexity}, \texttt{data\_flow\_fan\_out}, \texttt{sanitizer\_coverage}, \texttt{interprocedural\_depth})
    \item \textbf{Stage 3 results}: \texttt{LLMValidation} (\texttt{AttackerVerdict}, \texttt{DefenderVerdict}, \texttt{consensus\_verdict}, \texttt{consensus\_confidence}, \texttt{cvss\_base\_score}, \texttt{cvss\_vector}, \texttt{evidence\_narrative})
    \item \textbf{Stage 4 results}: \texttt{fused\_score}, \texttt{verdict}, \texttt{stage\_resolved}, \texttt{nl\_explanation}, \texttt{remediation}
\end{itemize}

The \texttt{ScanResult} model aggregates all findings and tracks cascade statistics: \texttt{resolved\_at\_sast}, \texttt{resolved\_at\_graph}, \texttt{resolved\_at\_llm}, and \texttt{unresolved}. The \texttt{cascade\_efficiency} property computes the fraction resolved at Stage~1, which is the primary efficiency metric.

Computed properties on the \texttt{Finding} model---\texttt{is\_escalated\_to\_graph} (checks $U \geq 0.5$) and \texttt{is\_escalated\_to\_llm} (checks whether the conformal prediction set is ambiguous)---provide convenience accessors that mirror the routing logic without duplicating it. The \texttt{display\_summary} property produces a one-line summary suitable for console output.

\subsection{Pipeline Orchestration}

The \texttt{PipelineOrchestrator} class (\texttt{src/orchestrator/pipeline.py}) coordinates the async execution of all four stages. It accepts a scan target (file path, directory, or GitHub URL), detects languages, and dispatches findings through the cascade. Stage~3 uses Python's \texttt{asyncio} to run attacker and defender agents concurrently (or sequentially under rate limits). The orchestrator tracks timing per stage and records \texttt{CascadeStats} for the final report.

\vspace{1em}

The methodology described in this chapter defines a complete pipeline from source code to triaged vulnerability report. Chapter~4 evaluates this pipeline experimentally: we measure cascade efficiency (what fraction of findings each stage resolves), classification accuracy (precision, recall, F1 against labeled datasets), conformal coverage (does the 90\% guarantee hold empirically), and end-to-end comparison against standalone tools (Semgrep, CodeQL, Snyk, LLM-only).
